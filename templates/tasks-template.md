# Задачи: [FEATURE NAME]

**Входные данные**: Проектные документы из `/specs/[###-feature-name]/`
**Предварительные требования**: plan.md (обязательно), spec.md (обязательно для пользовательских историй), research.md, data-model.md, contracts/

**Тесты**: Приведённые ниже примеры включают задачи тестирования. Тесты НЕОБЯЗАТЕЛЬНЫ — включайте их только если они явно запрошены в спецификации фичи.

**Организация**: Задачи сгруппированы по пользовательским историям для обеспечения независимой реализации и тестирования каждой истории.

## Формат: `[ID] [P?] [Story] Описание`

- **[P]**: Можно выполнять параллельно (разные файлы, нет зависимостей)
- **[Story]**: К какой пользовательской истории относится задача (например, US1, US2, US3)
- Включайте точные пути к файлам в описания

## Соглашения о путях

- **Единый проект**: `src/`, `tests/` в корне репозитория
- **Веб-приложение**: `backend/src/`, `frontend/src/`
- **Мобильное**: `api/src/`, `ios/src/` или `android/src/`
- Показанные ниже пути предполагают единый проект — скорректируйте на основе структуры из plan.md

## Зависимости и порядок выполнения

### Зависимости между фазами

- **Настройка (Фаза 1)**: Нет зависимостей — можно начинать немедленно
- **Фундамент (Фаза 2)**: Зависит от завершения Настройки — БЛОКИРУЕТ все пользовательские истории
- **Пользовательские истории (Фаза 3+)**: Все зависят от завершения Фундамента
  - Пользовательские истории могут выполняться параллельно (при наличии ресурсов)
  - Или последовательно в порядке приоритета (P1 → P2 → P3)
- **Полировка (Финальная фаза)**: Зависит от завершения всех нужных пользовательских историй

### Зависимости между пользовательскими историями

- **Пользовательская история 1 (P1)**: Может начаться после Фундамента (Фаза 2) — нет зависимостей от других историй
- **Пользовательская история 2 (P2)**: Может начаться после Фундамента (Фаза 2) — может интегрироваться с US1, но должна тестироваться независимо
- **Пользовательская история 3 (P3)**: Может начаться после Фундамента (Фаза 2) — может интегрироваться с US1/US2, но должна тестироваться независимо

### Внутри каждой пользовательской истории

- Тесты (если включены) ДОЛЖНЫ быть написаны и НЕ ПРОХОДИТЬ до реализации
- Модели до сервисов
- Сервисы до конечных точек
- Основная реализация до интеграции
- История завершена до перехода к следующему приоритету

### Возможности для параллельного выполнения

- Все задачи Настройки с меткой [P] могут выполняться параллельно
- Все задачи Фундамента с меткой [P] могут выполняться параллельно (в рамках Фазы 2)
- После завершения Фундамента все пользовательские истории могут начаться параллельно (при наличии ресурсов команды)
- Все тесты для пользовательской истории с меткой [P] могут выполняться параллельно
- Модели внутри истории с меткой [P] могут выполняться параллельно
- Разные пользовательские истории могут разрабатываться параллельно разными членами команды

## Стратегия реализации

### MVP в первую очередь (только Пользовательская история 1)

1. Завершить Фазу 1: Настройка
2. Завершить Фазу 2: Фундамент (КРИТИЧНО — блокирует все истории)
3. Завершить Фазу 3: Пользовательская история 1
4. **ОСТАНОВИТЬСЯ И ПРОВЕРИТЬ**: Протестировать Пользовательскую историю 1 независимо
5. Развернуть/продемонстрировать при готовности

### Поэтапная поставка

1. Завершить Настройку + Фундамент → Фундамент готов
2. Добавить Пользовательскую историю 1 → Тестировать независимо → Развернуть/Демонстрировать (MVP!)
3. Добавить Пользовательскую историю 2 → Тестировать независимо → Развернуть/Демонстрировать
4. Добавить Пользовательскую историю 3 → Тестировать независимо → Развернуть/Демонстрировать
5. Каждая история добавляет ценность, не ломая предыдущие

### Параллельная стратегия команды

При наличии нескольких разработчиков:

1. Команда совместно завершает Настройку + Фундамент
2. После завершения Фундамента:
   - Разработчик A: Пользовательская история 1
   - Разработчик B: Пользовательская история 2
   - Разработчик C: Пользовательская история 3
3. Истории завершаются и интегрируются независимо

## Примечания

- Задачи [P] = разные файлы, нет зависимостей
- Метка [Story] связывает задачу с конкретной пользовательской историей для отслеживания
- Каждая пользовательская история должна быть независимо завершаемой и тестируемой
- Убедитесь, что тесты не проходят до реализации
- Создавайте коммит после каждой задачи или логической группы
- Останавливайтесь в каждой контрольной точке, чтобы независимо проверить историю
- Избегайте: расплывчатых задач, конфликтов одного файла, межсторийных зависимостей, нарушающих независимость
